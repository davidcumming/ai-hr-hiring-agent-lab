# Copilot Studio Tool Readiness - Candidate Evaluation Council API

Status: **current lab contract plus partial manual Copilot Studio
configuration evidence.** The repository provides the source API contract,
the curated Copilot Studio / Power Platform custom-connector Swagger artifact,
and note-based evidence for one synthetic Copilot Studio topic workflow. It
does not provide source-controlled Copilot ALM export, durable
screenshot/export/transcript evidence, Entra app registration, production
identity, live Foundry execution, or production-ready Copilot integration.

## What the implemented contract provides

The committed source API contract is `openapi/evaluations-api.json`, generated
from the live app factory by `scripts/export_openapi.py` (drift-checked by
test). It remains OpenAPI 3.1 and is not replaced by the Copilot registration
artifact.

The curated registration artifact is
`openapi/copilot-studio/evaluations-tool.swagger.json`, generated by
`scripts/export_copilot_openapi.py`. It is Swagger 2.0 for Copilot Studio /
Power Platform custom connector compatibility and intentionally exposes a
fixture-reference-only submit shape for the lab. It must not be used for real
applicant data.

| Readiness item | Status in the implemented API |
|---|---|
| Stable operation IDs | `submitEvaluation` (`POST /api/evaluations`), `getEvaluation` (`GET /api/evaluations/{evaluation_id}`), `retrieveEvaluationForCopilot` (`POST /api/evaluations/retrieve`) |
| Canonical explicit-ID read | `getEvaluation` remains available for clients that can reliably supply the path parameter `evaluation_id`. |
| Copilot-friendly retrieve read | `retrieveEvaluationForCopilot` accepts a JSON body with exactly `evaluation_id`; it returns the same persisted audit-record envelope as `getEvaluation`. |
| Tool-suitable descriptions | Operation `summary`/`description` fields state purpose, advisory-only boundary, status vocabulary, and HTTP mapping in business language. The body-based retrieve description names the stored topic-variable binding intent. |
| Curated Swagger 2.0 registration artifact | `openapi/copilot-studio/evaluations-tool.swagger.json` with placeholder host `function-app-host.example`, `basePath: /`, `https` only, and exactly three actions. |
| Fixture-reference-only submit request shape | The curated Copilot artifact exposes `position_id`, `candidate_ref`, optional `idempotency_key`, optional `evaluation_question`, and optional `requested_rigor`; inline applicant text is not exposed in the Copilot registration artifact. |
| Body-based retrieve request shape | `EvaluationRetrieveRequest` exposes only body field `evaluation_id`. |
| Function key auth representation | The curated Swagger declares a header API key security definition named `x-functions-key`; the key value is supplied only through secure Copilot Studio / Power Platform connection settings. |
| Temporary lab identity headers | The curated Swagger exposes `X-Lab-Actor-Id`, `X-Lab-Roles`, and optional `X-Lab-Actor-Display` as visible/testable header parameters; these are simulated lab auth only, not Entra identity. |
| Standard response envelope | Adopted envelope subset: `status`, `evaluation_id`, `case_id` (always null - case-less), `correlation_id`, `user_message`, `safe_details`, `result`, `errors`, `warnings`. |
| Fixed status vocabulary | Emitted: `completed | blocked | validation_failed | unauthorized`; declared-reserved: `needs_input`, `error`. Business outcomes return HTTP 200; malformed body 400; missing identity 401; wrong role 403. |
| Correlation ID header | `X-Correlation-Id` accepted on submit, body retrieve, and GET requests and returned on responses that have a correlation id to carry. |
| Idempotency key header | `Idempotency-Key` accepted on `submitEvaluation` as an equivalent of the body field `idempotency_key` (one required; mismatch -> HTTP 400); replays return the original result without re-running the council. |
| No backend selection in requests | Request schemas contain **no** provider/model/deployment/endpoint/agent field; provider selection is server-side configuration only. |
| Advisory-only output | Every result carries `decision_support_only: true` and `human_review_required: true`; recommendation labels are the fixed advisory enum; `effective_rigor` and the escalation-policy outcome are in the result. |
| Synthetic examples only | All fixtures and documented examples are synthetic (`cand-sample-001`, `pos-sample-001`). |

## Manual lab workflow

One manual Copilot Studio lab topic workflow is configured and note-evidenced:

| Workflow element | Current state |
|---|---|
| Topic | `E6 Evaluate Sample Candidate` in the `HR Hiring Agent Lab` Copilot Studio agent, environment `CHI-LAB-SANDBOX`. |
| Submit action | Calls `submitEvaluation` for the synthetic fixture request. |
| Stored variable | Stores the returned `evaluation_id` in `submitted_evaluation_id`. |
| Retrieve action | Calls `retrieveEvaluationForCopilot` with `submitted_evaluation_id` bound to body field `evaluation_id`. |
| Tool availability | `submitEvaluation` and `retrieveEvaluationForCopilot` are set to `Only when referenced by topics or agents` so the topic owns the stateful workflow. |
| Observed result | The topic completes for the prompt `Evaluate the sample candidate.`, displays the stored id, and returns an advisory/audit summary for the same id. |
| Evidence depth | Partial note-based evidence plus chat-attached screenshot context; no durable screenshot, connector export, topic export, transcript, or automated replay artifact is stored in the repo. |

The workflow does not prove a multi-candidate case workflow, arbitrary
document intake, production identity, Entra authorization, live Foundry
execution, or production readiness.

## Manual registration guide

Use
[`copilot-studio/registration-guide.md`](./copilot-studio/registration-guide.md)
for the human portal checklist and hosted smoke commands. The guide includes
both paths for the temporary simulated lab identity headers:

- Preferred lab path: configure static/default/internal `X-Lab-*` headers if
  the Copilot Studio or connector UI supports them.
- Fallback path: import and review the actions, but do not claim end-to-end
  Copilot invocation until connector policy/header support is configured or a
  later Entra-auth slice replaces simulated headers.

## Not yet in place

- Entra app registration and delegated OAuth: authentication is still
  **simulated lab identity headers** (`X-Lab-Actor-Id`, `X-Lab-Roles`) - a lab
  stand-in, never an Entra substitute.
- Source-controlled Copilot Studio ALM export or unpacked topic/connector
  configuration. The portal configuration is manual and can drift.
- Durable screenshot/export/transcript evidence for the manual portal
  workflow. Current evidence is note-based.
- Connector/DLP decisions per the target architecture: REST/API tool
  registration, premium/custom connector implications, and production
  governance remain to be re-validated when live wiring is undertaken.
- Live provider wiring, live Foundry model execution, production identity,
  real applicant data, multi-candidate case workflow, and final hiring
  decision authority.

## Tool description drafts

- **submitEvaluation** - "Submit one synthetic candidate evaluation to the
  advisory Calibrated Evaluation Council for a sample position and approved
  rubric. Returns an advisory, evidence-grounded result that always requires
  human review. Never makes or implies a hiring decision."
- **getEvaluation** - "Retrieve the full persisted audit record of a prior
  evaluation by evaluation_id, including every council role output, rigor
  resolution, escalation triggers, and quality-gate results. Advisory decision
  support only."
- **retrieveEvaluationForCopilot** - "Retrieve the full persisted audit record
  of a prior evaluation by body field evaluation_id. Use this action when a
  Copilot Studio topic stores an evaluation id in a workflow variable and must
  bind that value into the retrieve request. Advisory decision support only."
