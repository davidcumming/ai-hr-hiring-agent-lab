# 40.2 Technical Architecture Guidelines

## Microsoft-Stack Guardrails and Service Mappings for Agentic Capabilities

> **Status:** Canonical technical architecture guidelines. Binding on every agentic capability built on this stack.
> **Purpose:** Define the Microsoft-stack guardrails and the service-to-responsibility mapping so that every agentic capability — HR Hiring, Communications Review, Meeting Notes, and any future capability — is built the same safe, auditable, testable way without re-inventing its architecture.
> **Scope:** Capability-to-stack guidance. This document does not specify any single domain product. Domain specifics (topics, prompts, gates, evidence names, test data) are supplied by each capability's own specialization overlay (Section 16).
> **Authority:** Where this document commits to specific products (Cosmos DB for memory, PostgreSQL for RAG, Azure Storage for system of record), those commitments are binding. On any conflict about *principles* — advisory-only AI, the facade owning controls, memory subordination — the principles stated in Section 1 govern.

---

## 0. How to use this document

Read this before building any new agentic capability. The default structure for any capability is "the same way as every other capability in this family": vary only the domain-specific parts named in the specialization overlay; keep the shared architecture identical.

The guardrails fall into three groups:

1. **Invariants** (Sections 1–2): the binding principles and the request flow that never change between capabilities.
2. **Service guardrails and mappings** (Sections 3–16): for each architectural concern, which Microsoft service implements it, the boundary that binds it, and what it must never do. These cover all required areas: the Copilot front door, Foundry, Power Platform, the Azure control plane, memory, RAG, workflow state, storage, API/tool integration, identity and access control, security and privacy, auditability, observability, data residency, and manual-configuration policy.
3. **Applying the guidance** (Sections 17–19): the shared contract, the data-placement decision guide, and the new-capability checklist.

## 1. Target stack and binding principles

### 1.1 The stack at a glance

```text
Microsoft Entra ID (identity boundary)
  -> Copilot Studio (conversation, orchestration, tool/action selection)        [FRONT DOOR]
      -> REST/API tool/action (OpenAPI + Entra auth)                            [TOOL BOUNDARY]
          -> Azure Functions / thin API facade (deterministic business contract) [CONTROL PLANE]
              -> Azure Storage: Blob + Table (system of record + evidence ledger)
              -> Azure Key Vault (secrets)
              -> Memory Service / Memory SDK Adapter (contextual memory contract)
                   -> Cosmos DB (memory persistence store)
              -> PostgreSQL + pgvector (RAG / retrieval store)
              -> AI backends (advisory only, invoked by the facade):
                   Microsoft Foundry model endpoint  (single structured inference)
                   Foundry Agent Service / Agent Framework 1.0 workflow (multi-step / multi-agent)
              -> Application Insights / Log Analytics (operational telemetry)
```

The single most important shape in this architecture: **Copilot Studio decides what to call next; the Azure facade decides whether the action is allowed.** Every other guardrail follows from that separation.

### 1.2 Binding principles

These apply to any agentic capability built on this stack, without exception.

1. **The chat surface is not the workflow engine.** Copilot Studio guides the user and selects tools; the Azure facade decides whether an action is permitted.
2. **The model is not the authority.** Any AI backend may draft, classify, summarize, retrieve, or evaluate, but it never approves, rejects, authorizes, mutates state, or finalizes a business decision.
3. **Tool results are rendered, not re-decided.** When the facade returns `blocked`, `unauthorized`, or `validation_failed`, the agent displays that outcome and does not override, soften, or escalate it.
4. **User- and applicant-supplied content is untrusted input.** It is passed as typed payload to tools. It does not steer orchestration, alter prompts, change gates, or grant authority.
5. **State-changing operations require evidence.** If business state changes, a business-readable evidence event is written.
6. **Memory and RAG are context, not authority.** They improve continuity, drafting, retrieval, and explanation, but they are always subordinate to current case state, artifacts, approvals, evidence, human-review records, and workflow gates.
7. **Humans make business-finalization decisions.** No AI output is official until a human reviews and approves it.
8. **Every capability is testable deterministically — and evaluated live.** Every AI-powered operation has a deterministic mock equivalent so gates, authorization, and contracts are provable without the live model. Deterministic tests are necessary but not sufficient: agentic behaviour also requires live-model evals per the process doc (§18–§19). Eval scenarios use synthetic or de-identified data by default; production or per-subject personal data enters an eval only under documented eval-data-governance approval.
9. **No silent premium licensing.** Any path requiring a premium/custom connector, premium Power Automate, Microsoft 365 Copilot licences, or provisioned throughput stops the build until explicitly approved.
10. **No silent manual configuration.** Infrastructure, identity, network, and security configuration is declared and reviewable, not hand-applied in a portal without record (Section 14).

### 1.3 Division of responsibility

| Layer | Owns | Must not own |
|---|---|---|
| Copilot Studio | Conversation, topic flow, slot-filling, tool/action selection, rendering results | Business-rule authority, RBAC, gates, approval, evidence, trusted state, direct AI-backend calls |
| Azure Functions / API facade | Identity validation, authorization, gates, input validation, idempotency, persistence, evidence, response envelope, the decision whether/which AI backend to call | Conversation flow, topic routing |
| Azure Storage (Blob + Table) | System of record: case state, artifacts, evidence payloads, metadata, idempotency, case ACLs | Conversation orchestration; deferring authority to a model |
| Memory Service / Memory SDK Adapter | Memory contract: scoped retrieval, save triggers, the proposed/confirmed/superseded/restricted/expired memory lifecycle, context-pack construction, provenance/status enforcement | System of record, evidence ledger, approval, workflow gates |
| Cosmos DB (memory persistence store) | Persistence for governed memory records and memory indexes, accessed only through the memory service/facade | Direct business authority, direct application/workflow access, approval, gates, evidence |
| PostgreSQL + pgvector (RAG store) | Indexed reference material and embeddings for retrieval | System of record, authority over generated output, evidence |
| Microsoft Foundry model endpoint | Single, stateless, structured inference when the facade asks | State mutation, approval, persistence, authority |
| Foundry Agent Service / Agent Framework 1.0 workflow | Multi-step, tool-using, or multi-agent inference/evaluation when the facade asks; returns proposed structured output with trace/eval metadata | State mutation, approval, becoming the workflow engine or system of record |
| Key Vault | Secrets and connection material | Business logic |
| App Insights / Log Analytics | Operational telemetry | Canonical audit |
| Evidence ledger (in Azure Storage) | Canonical, business-readable audit history | Runtime monitoring dashboard |

**Rule:** no business-rule code lives in Copilot Studio; no conversational orchestration lives in the Azure services.

## 2. The reusable request flow

Every capability moves a request through the same path. This is a generic flow, not implementation code.

```text
User conversation in Copilot Studio
  -> topic/prompt interprets intent; structured card collects typed input if needed
    -> agent selects and calls a REST/API tool/action
      -> Azure facade receives the typed request
          1. validate input schema
          2. validate actor identity (Entra claims, not free-text fields)
          3. authorize (global role + case scope)
          4. check the workflow gate
          5. enforce idempotency
          6. [if context needed] retrieve scoped memory context pack through the
                Memory Service / SDK Adapter (backed by Cosmos DB)
                and/or retrieve reference passages (Postgres + pgvector)
                -- status/scope/provenance labelled; never authoritative
          7. [only if inference needed] call ONE AI backend:
                Microsoft Foundry model endpoint  OR  Foundry Agent Service / Agent Framework 1.0 workflow
          8. validate the structured AI output against its schema
          9. write artifact / evidence payloads to Blob
         10. commit state change + evidence event metadata together
                (same-partition entity-group transaction where possible;
                 never report `completed` until both are durable)
         11. [if a save trigger applies] extract proposed memory candidates; the
                Memory Service / SDK Adapter applies service rules
                (scope, status, provenance) before any write to Cosmos DB
         12. emit safe telemetry (App Insights)
      -> facade returns the standard response envelope
    -> Copilot Studio renders the result (completed / blocked / needs_input /
       validation_failed / unauthorized / error)
  -> agent explains the outcome and asks the next useful question
```

Read-only status questions ("where are we?", "what can I do next?", "why is this blocked?") follow the same shape: the agent calls a status/evidence service and answers from the returned state, never inferring business state from conversation memory. **No state change ever happens on a path that bypasses the agent and the Azure facade.**

## 3. Copilot front door and orchestration

**Implemented with:** Copilot Studio (pay-as-you-go agent in a Power Platform sandbox). It owns conversation and orchestration — topics, trigger phrases, slot-filling, guided choices, tool/action selection. Authored topics are the backbone; generative orchestration is the selector, not the decision-maker.

**Guardrails**

- Follow the documented domain process. Use clear trigger phrases and slot-filling for required inputs.
- Never ask a bare open-ended question when the process structure is known. Every user-facing question offers options with a recommended choice, a sensible default, or a suggested draft the user can edit.
- Render facade outcomes verbatim in business language. A `blocked` result is process guidance, not a system error.
- Hold only transient active-turn values in conversation variables. Durable state lives in Azure Storage; durable memory lives in the Cosmos-backed memory service behind the facade.

**Must not:** own business rules, RBAC, gates, approvals, evidence, or trusted state; call any AI backend directly; re-decide a facade outcome; serve as the system of record.

## 4. API / tool integration pattern

**Implemented with:** Copilot Studio REST/API actions registered with OpenAPI + Entra authentication. No premium connectors and no custom connectors as the default path.

A tool is the *only* path to business-state change: a thin, agent-callable layer over one Azure facade endpoint that declares an input schema, passes the typed payload through, and returns the structured envelope. Tools are named after the business action they perform (`draft_document`, `approve_artifact`, `export_artifact`).

**Guardrails**

- Prove the integration path before building any domain workflow: a signed-in Copilot Studio user invokes a secured Azure endpoint under the pay-as-you-go / no-premium / no-custom-connector pattern, the facade derives identity from trusted claims, returns the standard envelope, and writes an evidence event. This is the non-negotiable integration-validation gate.
- **The agent is under source control before the gate passes.** A hard precondition of the integration-validation gate is that the Copilot Studio agent **is in a solution and committed via Git integration** — not exported ad hoc, not configured only in the portal. Use the concrete Copilot Studio ALM mechanisms: **solution-based ALM** (the agent and its components live in a Power Platform solution), **native Git integration** (the solution is connected to a Git repository), **Power Platform pipelines** (for promotion across environments), and the **`microsoft/copilot-alm-starter`** GitHub Actions starter for CI/CD. An agent that exists only as unversioned portal configuration is manual-config debt (§16) and fails the gate.
- Structured cards are input surfaces only. A card must not call a mutation endpoint from the browser, must not contain client-side logic that re-orders steps, and must not pre-fill business facts the user has not supplied.
- Every tool maps to exactly one facade endpoint and returns the shared response envelope (Section 15).

**Must not:** become a place where workflow logic or sequencing lives; introduce a premium or custom connector silently to bridge an integration gap.

## 5. Azure service boundaries (the control plane)

**Implemented with:** Azure Functions or a thin Azure API endpoint — the facade. This is where all deterministic logic lives; the facade runs the full request flow (Section 2) for every request, and is the **only** layer that calls an AI backend, retrieves memory, or queries the RAG store — and only after its deterministic controls have run.

**Guardrails**

- Authorize before mutating. Validate before mutating. A state change and its evidence event are one logical unit: write evidence metadata and the state entity in the same Table entity-group transaction (same table, same case-ID partition) wherever possible; where a payload Blob is involved, write the Blob first, then commit state + evidence metadata transactionally. An operation returns `completed` only after both are durable. If the evidence write fails after the state write, the facade returns `error` and the idempotent retry or a reconciliation sweep must complete the evidence event before the case can advance. A state change with no recoverable evidence path is a defect (§20).
- Mutating calls require idempotency keys.
- The facade decides *whether* an action is allowed and *whether/which* AI backend to call. Copilot Studio only decides what to call next.
- Heavy work (for example complex PDF rendering) escalates to Azure Container Apps rather than to premium Power Automate.
- **Long-running work runs behind the facade, not in the conversation turn.** A multi-step Agent Framework 1.0 / Foundry Agent Service workflow, or a 429-heavy PAYG endpoint with backoff (§6), can exceed the Copilot Studio connector / tool-call turn timeout. For these, the facade implements the long-running pattern with **Azure Durable Functions** (orchestrator + activity functions for the multi-step work) **or the Foundry Agent Service async run API**: the synchronous call validates, gates, idempotency-checks, starts the durable orchestration (or async run), and returns immediately with an `in_progress` status and an operation ID; a separate status-poll tool reads progress and returns the final envelope once the work is durable. The wire-level status vocabulary (`in_progress`), the `operation_id` field, and the status-poll contract are defined in §17 (the shared contract) of this document; the process-level rule that agents must treat `in_progress` as non-terminal and poll rather than block is Process Doc §20.1. This section specifies only the facade-side implementation. The deterministic controls still run *before* the orchestration starts, and state/evidence are still committed transactionally (above) when the work completes — never on the polling read.
- **API Management is the growth path.** With one or two capabilities, per-capability Function facades with Entra auth are right-sized; as the capability count grows, place **Azure API Management** in front of the per-capability Function facades to standardize authentication, throttling, versioning, and the OpenAPI surface Copilot Studio consumes. This is the intended evolution, not a redesign — the facade keeps owning the controls behind APIM.

**Must not:** run the conversation, route topics, or make free-form UX decisions; defer a control decision to a model.

## 6. Foundry usage boundaries

**Implemented with:** Microsoft Foundry model endpoint *or* Foundry Agent Service / Microsoft Agent Framework 1.0 workflow. Always behind the facade. Always advisory. The facade chooses the **smallest** backend that meets the need.

> **Platform generation (as of Build 2026).** Azure AI Foundry is now **Microsoft Foundry**. **Microsoft Agent Framework 1.0 reached GA at Build 2026** — the production successor to the Semantic Kernel / AutoGen patterns, with skills, memory, and middleware as first-class concepts. The **next-generation Foundry Agent Service is GA**, built on the OpenAI Responses API. New capabilities targeting a multi-step or multi-agent backend build on **Agent Framework 1.0 workflows** running on (or callable from) the Foundry Agent Service. The exact generation each new capability pins is recorded in an ADR under `Initial_Documentation/adr/` (being drafted) so the choice is explicit and re-eval triggers (process doc §21.1) can key off it.

| Signal | Microsoft Foundry model endpoint | Foundry Agent Service / Agent Framework 1.0 workflow |
|---|---|---|
| Single structured generation/classification | Yes (default) | Overkill |
| Tool/function use during inference | No | Yes |
| Multi-step reasoning | No | Yes |
| Multi-agent propose/validate or rubric evaluation | No | Yes |
| Retrieval/RAG over reference material | Borderline — prefer workflow | Yes |
| Reusable AI capability needing tracing/eval | No | Yes |

Default to a **Microsoft Foundry model endpoint** for a single bounded structured generation or classification with no tool use, multi-step reasoning, or second-model validation. Use the **Foundry Agent Service or an Agent Framework 1.0 workflow** when the task genuinely needs orchestration the facade should not hand-roll: multi-step reasoning, tool use during inference, retrieval over reference material, multi-agent propose/validate, or a reusable capability that benefits from production-grade tracing and evaluation. The decision logic above is unchanged by the generation refresh — only the backend that implements the right-hand column has moved to Agent Framework 1.0.

**Guardrails (either backend)**

- The facade owns prompt construction, the backend call, output validation, retry, persistence, and evidence.
- The backend returns proposed structured output (plus trace/eval metadata for multi-step runs); it never mutates state and is never official until a human approves it.
- Allow one bounded corrective retry on schema-validation failure, then stop and ask for human correction. Never silently accept invalid output.
- Distinguish validation retries from transient-fault retries. On throttling (429) or transient backend failure, apply bounded exponential backoff honouring `Retry-After`, within a per-request retry budget sized to the conversation-turn timeout; PAYG endpoints without provisioned throughput must treat 429 as a normal expected condition. If the budget is exhausted, return `error`, persist nothing, and leave the operation safely re-invokable under its idempotency key. Never return partial or unvalidated output in place of a failed call.
- Record per-generation metadata: operation/correlation/actor/case IDs, prompt/template ID and version, backend type, model/agent/workflow ID and version, trace/eval IDs (multi-step), input/output schema versions, source refs, validation result, retry count, token/latency metrics, artifact ID/version, evidence event ID.
- Prompts are versioned and approved before use.
- No provisioned throughput for Foundry endpoints or agents without explicit approval; no always-on AI/agent runtime without justification.

**Must not:** be called directly by Copilot Studio for any state-changing or business-affecting generation; mutate state, approve, or persist anything; become the workflow engine or system of record; access Cosmos memory or the RAG store directly for business-affecting operations.

### 6.1 Eval execution and artifact store

**Foundry projects are the default eval execution environment and artifact store.** Foundry **tracing and evaluations are GA** (any-framework support in preview), including **trace-based evaluation of real runs**, and the **`azure-ai-evaluation` SDK** covers programmatic scenario runs with built-in and custom rubric evaluators. Using a Foundry project as the eval home gives run records, model/prompt/agent version capture, dashboards, and access control inside the Azure tenant and residency boundary (§15) without a hand-rolled harness.

Consequently, the `live-eval-runner` skill **reduces to orchestrating the `azure-ai-evaluation` SDK and exporting a repo-safe summary** (scenario IDs, pass/fail against the threshold table, version manifest, run/trace IDs — no sensitive payloads). Hand-roll only what the **pass-threshold contract** requires and Foundry does not provide (e.g. the capability's specific risk-tiered threshold table and pass/fail roll-up from the process doc §19). Eval artifacts honour residency and access control per §15.

## 7. Power Platform usage boundaries

**Implemented with:** Power Platform sandbox hosting the Copilot Studio agent only — and nothing more; all business logic, state, identity enforcement, and AI calls live in Azure behind the facade.

**Guardrails**

- The Copilot Studio agent runs pay-as-you-go in a sandbox environment with standard (non-premium) connectors only.
- Power Automate, if used at all, is limited to non-business-critical, non-gating convenience automation and never to enforce a control, finalize a decision, or move business state. Gates and approvals are enforced in the facade, not in a flow.
- Dataverse and other Power Platform data stores are **not** the system of record, the evidence ledger, the memory store, or the RAG store. Authoritative data lives in Azure (Sections 8, 9, 10, 11).

**Must not:** use premium Power Automate, premium connectors, custom connectors (default path), Microsoft 365 Copilot licences, or Microsoft 365 Business Premium without explicit approval; place business rules, RBAC, gates, or evidence in Power Platform; let a Power Automate flow perform a state-changing business action outside the facade.

## 8. Storage pattern (system of record)

**Implemented with:** Azure Storage — Table for state, Blob for payloads. Azure Storage is the authoritative system of record; Cosmos DB and PostgreSQL are deliberately *not* (see §1.3 and the placement guide in Section 18).

| Concern | Store |
|---|---|
| Case / work-item state, idempotency records, case ACLs | Azure Table Storage |
| Source documents, normalized documents, generated drafts, exports, evidence payloads | Azure Blob Storage |

Partition Table data by case ID. Use deterministic Blob paths that are never overwritten; every saved draft or revision is preserved as a versioned artifact. The official artifact is the version recorded as approved by the relevant approval evidence event.

**Guardrails**

- Update the state entity with ETag / optimistic concurrency; surface a stale-version conflict as a recoverable error (re-read, retry).
- If a Blob write succeeds but the Table write fails, reconciliation identifies and handles orphan payloads.
- The system of record and the evidence ledger must be recoverable. Azure Table Storage has no point-in-time restore: each capability declares a backup mechanism (e.g. scheduled export of Table state and evidence metadata to versioned, immutable Blob, and/or GZRS/RA-GZRS with a documented failover decision), an RPO/RTO, and a periodic restore test. Blob containers holding artifacts and evidence payloads enable versioning and soft delete from slice 1; evidence containers additionally carry **time-based retention (an immutability policy) as the lab default** (§13), with legal-hold/locked-WORM documented per capability overlay; resource locks prevent accidental account deletion. A capability with no tested recovery path for state and evidence is not production-ready, and any geo-secondary must stay within the approved residency boundary (§15).
- Documents follow a uniform lifecycle: store the original in Blob, normalize once to Markdown + YAML metadata at ingestion (never reconvert repeatedly), use the normalized version for processing, generate strict JSON, render JSON into a branded template, export DOCX/PDF, and write evidence. The model never produces the official final document directly.
- Exported artifacts carry visible metadata: case ID, artifact type and version, generated/approved timestamps, evidence event ID, an advisory/AI-use notice, and source/template references.

**Must not:** be replaced by Cosmos DB, PostgreSQL, Dataverse, or conversation state as the source of truth.

## 9. Workflow state pattern

**Implemented with:** facade-enforced state machine over Azure Table Storage; facade-enforced gates; Copilot Studio renders outcomes.

Workflow state is authoritative and lives in the system of record. The facade reads current case state, evaluates the gate, and either performs the action (advancing state with optimistic concurrency and writing evidence) or returns `blocked` with a business-language reason the agent renders verbatim.

**Guardrails**

- Each state-changing action is gated deterministically in the facade. A gate is code that refuses the action, not prompt text. "Only approve if both reviewers agree" is enforced by the service that refuses the action.
- Approval is a deterministic, gated, evidenced action in the facade — never something the agent or model finalizes. AI output is advisory until a human reviews and approves it; the approved, versioned artifact is the official one.
- The agent performs no autonomous distribution or irreversible external action. Any sharing or release beyond producing an exportable artifact is itself a gated, evidenced, human-controlled step.
- State transitions are recorded as evidence (Section 11). Read-only "where are we / what's next / why blocked" queries are answered from authoritative state via a status/evidence service.

**Must not:** rely on conversation memory or Cosmos memory for trusted state, candidate status, or gate decisions; allow a state change on any path that bypasses the facade.

## 10. Memory pattern (contextual support)

**Implemented with:** a facade-owned **Memory Service / Memory SDK Adapter** backed by **Cosmos DB** as the default memory persistence store.

> **Cosmos DB is the storage substrate, not the memory contract.** All memory retrieval, save, supersede, reject, restrict, expire, and context-pack construction operations go through the facade-owned memory service / SDK layer. Application code must not treat Cosmos DB itself as the memory contract.

Memory is a reusable contextual support layer (continuity, drafting, review, explanation), never a second source of truth (see §1.2 principle 6).

Common record types: **session summaries** (turn-level continuity), **durable facts** (confirmed contextual facts with source references), **proposed facts** (candidates awaiting confirmation), and **scoped context packs** (purpose-bound bundles returned to a tool or AI backend). Each item carries scope, status, source-event reference, evidence reference, artifact reference, and warnings.

**Status model (canonical enum):** `proposed`, `advisory`, `confirmed`, `rejected`, `superseded`, `restricted`, `audit-only`, `expired`. Rejected, superseded, restricted, audit-only, and expired memory is excluded from current drafting/assessment except for explicitly labelled historical explanation.

**What the Memory Service / SDK Adapter owns:** memory record schema enforcement; scope enforcement (user / job-case / candidate / process, plus the cross-cutting session view); status lifecycle enforcement across the canonical enum; save- and retrieval-trigger handling; context-pack construction; provenance and source-event references; candidate confirmation, rejection, and supersession; sensitive-memory restriction; expiry and retention handling; conflict handling when memory contradicts authoritative state (authoritative state wins); and emission of memory lifecycle audit events.

**Guardrails**

- Retrieval is purpose-bound and occurs only after identity, authorization, case scope, and workflow stage are known. Saving durable memory occurs only after the service classifies the source event, attaches provenance, and applies status/scope rules.
- Memory is scope-bound. Sensitive per-subject memory must never bleed across subjects, cases, jobs, or users; any cross-scope reuse requires a separately governed, de-identified, allowed path.
- An AI backend may receive a service-built context pack and may *propose* memory candidates, but only the governed memory service confirms, persists, supersedes, rejects, restricts, or expires durable memory.
- Memory service responses label memory as contextual, scoped, and non-authoritative.
- Memory lifecycle events (`memory_context_retrieved`, `memory_confirmed`, `memory_superseded`, `memory_expired`, etc.) are metadata-first audit records.
- Cosmos DB is the default committed memory persistence store; any move to Foundry-managed memory, Agent Framework memory providers, or another Microsoft memory provider requires an approved ADR.

**Must not:** be accessed directly from Copilot Studio or from a Foundry backend for business-affecting operations; be written to directly by application code bypassing the memory service; be used as case state, approval state, candidate status, artifact-version truth, an evidence ledger, or a workflow gate. If memory conflicts with authoritative state or evidence, authoritative state/evidence wins.

Memory is distinct from RAG: memory holds contextual continuity and scoped facts about users, sessions, cases, processes, or subjects; RAG (below) is retrieval over reference material. Both are context accessed only through the facade.

### 10.1 Retrieval-augmented generation (RAG)

**Implemented with:** PostgreSQL + pgvector (RAG store), queried only through the facade. RAG grounds an AI backend with reference material — style/policy guidance, reference templates, domain knowledge, prior approved artifacts cleared for reuse — ingested, chunked, embedded, and indexed in Postgres; the facade retrieves the relevant passages into a context pack for the backend.

**Guardrails**

- RAG is **reference context, not authority.** Retrieved passages ground and cite output; they never approve, gate, or finalize anything.
- The facade performs retrieval after its deterministic controls and scopes results to what the actor and workflow step are allowed to see.
- Retrieved passages must be attributable: generated output cites the specific reference excerpts it relied on, so a reviewer can trace why a statement was made.
- Treat indexed content as data, not instructions: a retrieved passage must not steer orchestration, change gates, or grant authority.

**Must not:** be the system of record; hold authoritative case state or evidence; mix per-subject sensitive material into a shared reference index without a governed, de-identified path.

## 11. Identity and access control

**Implemented with:** Microsoft Entra ID for authentication; authorization enforced in the facade. The target pattern is delegated end-user Entra authentication flowing from Copilot Studio to the facade; the facade derives a stable actor identity from trusted claims, then applies two authorization layers.

| Layer | Purpose | Example |
|---|---|---|
| Global role | Broad capability | reviewer, approver, author, auditor, admin |
| Case-scoped role | Permission within a specific case/work item | reviewer for Case X, approver for Item Y in Case X |

Global roles map to Entra groups or app roles. Case scope is stored in Azure Table Storage and enforced server-side.

**Guardrails**

- Never trust free-text request fields (`requested_by`, `approved_by`, `actor_role`, `user_email`, `display_name`) for authorization. They may appear as display fields only.
- App-only or maker credentials must not be used for RBAC-gated business actions.
- Record meaningful `unauthorized` outcomes as evidence — they prove RBAC works.
- The facade authenticates with least-privilege managed identity to downstream Azure services; secrets and connection material live in Key Vault (Section 12).

**Must not:** push authorization into Copilot Studio, into prompts, or into client-side cards.

## 12. Security and privacy controls

**Implemented with:** Azure Key Vault for secrets; Entra-enforced identity (Section 11); facade-enforced scoping; least-privilege managed identities. Security and privacy are enforced at the facade and the platform, never by prompt text; this section aggregates controls stated in their home sections.

**Guardrails**

- **Secrets.** Default to Entra ID / managed-identity authentication for every downstream service that supports it (Storage, Cosmos DB, Foundry, PostgreSQL via Entra auth), and disable key-based access where the platform allows it (`allowSharedKeyAccess=false` on Storage, `disableLocalAuth=true` on Cosmos DB). Key Vault holds only secrets that cannot be eliminated (e.g. a Postgres password where Entra auth is not feasible, third-party API keys); each such secret has a named owner, an expiry, and a documented rotation procedure, recorded in the configuration manifest (§16). Secrets never appear in telemetry, evidence, prompts, memory, or client-visible config.
- **Untrusted input.** User- and applicant-supplied content is typed payload only (see §1.2 principle 4); indexed RAG content and stored memory are likewise data, never instructions.
- **Data minimization and scoping.** Memory and case data are scope-bound; sensitive per-subject material must not bleed across subjects, cases, jobs, or users, and cross-scope reuse requires a separately governed, de-identified, allowed path (see §10, §11).
- **Least privilege.** Each component holds only the access it needs; maker/app-only credentials are not used for RBAC-gated business actions.
- **No sensitive data in logs.** Full source documents, full generated drafts, personal/applicant data, secrets, and raw prompt/agent/workflow content are never written to telemetry unless explicitly approved (Section 13).
- **Human control of release.** No autonomous external distribution; any sharing beyond producing an exportable artifact is a gated, evidenced, human-controlled step.

**Must not:** store secrets outside Key Vault; rely on a prompt instruction as a security or privacy control; expose sensitive data in client-visible configuration, telemetry, or cross-scope memory.

## 13. Auditability

**Implemented with:** Azure Table (evidence metadata) + Azure Blob (evidence payloads) — the evidence ledger; the canonical, business-readable audit history, distinct from operational telemetry (Section 14). Every state-changing action emits a typed evidence event with event type, actor, role, case ID, timestamp, artifact version where applicable, operation/correlation IDs, and a structured payload. Evidence is immutable once written. Immutability is enabled **from slice 1**, not deferred to production-readiness: enable **Blob versioning + soft delete** on evidence containers from the first slice (already required, §8), and adopt **time-based retention (immutability policy) on evidence containers as the lab default**. The stricter **legal-hold / locked-WORM** decision is documented per capability overlay, but the baseline retention is on from the start — evidence written without immutability cannot be made immutable retroactively.

**Guardrails**

- Use business-readable event names in a consistent style (`document_uploaded`, `artifact_drafted`, `artifact_approved_by_<role>`, `artifact_export_generated`, `action_blocked`, `action_unauthorized`, `validation_failed`). Each capability supplies its own domain event names in the same style.
- Write evidence for meaningful `blocked`, `unauthorized`, and `validation_failed` outcomes — they prove gates and RBAC work. Do not log trivial typos as evidence.
- Memory lifecycle events use the same readable style but are audit/lifecycle records, never substitutes for workflow evidence.
- A correlation ID ties the conversation turn, tool call, evidence event, and telemetry together.

**Must not:** be replaced by free-text logs, by telemetry, by conversation memory, or by Cosmos/Postgres records; double as a runtime monitoring dashboard.

---

## 14. Observability and telemetry

**Implemented with:** Application Insights / Log Analytics. Telemetry is operational and distinct from the evidence ledger (Section 13); the canonical audit never lives only in telemetry.

**Safe to log:** request/correlation IDs, action name, status, latency, token counts, backend type, model/agent/workflow ID and version, trace/eval IDs, artifact IDs, evidence event IDs, safe error category.

**Never log:** full source documents, full generated drafts, personal/applicant data, secrets, or raw prompt/agent/workflow content unless explicitly approved.

Each capability should expose a simple, business-language case/work-item activity log in Copilot Studio with progressive disclosure: summary first, drill-down on request, technical IDs only when useful.

**Must not:** stand in for the canonical audit; contain sensitive payloads.

## 15. Data residency expectations

**Implemented with:** region-pinned Azure resources; documented region of record per capability. Because these capabilities may process personal and applicant data, residency is a first-class architectural concern.

**Guardrails**

- **Single declared region of record.** Each capability declares the Azure region in which its system of record (Azure Storage), evidence ledger, memory store (Cosmos DB), and RAG store (PostgreSQL) reside. The default expectation is that all data-bearing services for a capability are pinned to the same approved region (or regional pair for resiliency) unless an approved ADR states otherwise.
- **Keep data-bearing services co-located.** Storage, Cosmos DB, PostgreSQL, Key Vault, and the facade are provisioned in the declared region. Telemetry retention (App Insights / Log Analytics) is configured to the approved region and retention period.
- **Foundry endpoints honour residency.** The Microsoft Foundry model endpoint or Agent Service used by a capability must be in an approved region consistent with the data residency declaration; cross-region inference that would move personal data outside the approved boundary requires explicit approval.
- **No silent cross-border movement.** Replication, backup, disaster-recovery, or analytics paths that would move data across a residency boundary must be explicitly approved and recorded.
- **Residency is documented in the overlay.** Each capability's specialization overlay records the region of record, any approved exceptions, and the retention windows for state, evidence, memory, and telemetry.
- **Eval artifacts follow residency and access control.** Live-eval inputs, outputs, and transcripts may reproduce sensitive case content; they are stored in an access-controlled location inside the declared residency boundary, with their own retention window recorded in the overlay.

**Must not:** provision data-bearing services in ad-hoc regions; allow telemetry, backups, or AI inference to relocate personal data outside the approved boundary without a recorded decision.

## 16. Manual configuration policy references

**Implemented with:** declared, reviewable configuration (infrastructure-as-code where available); recorded exceptions for unavoidable manual steps. Manual, undocumented portal configuration is an anti-pattern here (see §1.2 principle 10).

**Policy references and guardrails**

- **Declare configuration, do not hand-apply it.** Resource provisioning, network rules, identity bindings, RBAC role assignments, Key Vault access policies, budgets, tags, and residency pinning are declared in reviewable form (IaC or a recorded configuration manifest), so each thin slice is reproducible and auditable.
- **Record unavoidable manual steps.** Where a setting can only be applied manually (for example a tenant-level Entra or Copilot Studio setting), record what was changed, by whom, when, and why, in the capability's configuration record.
- **No silent premium licensing (cost policy).** Any path requiring a premium/custom connector, premium Power Automate, Microsoft 365 Copilot licences, Microsoft 365 Business Premium, prepaid credit packs, or provisioned throughput (Foundry, Cosmos, Postgres) stops the build until explicitly approved. See the cost and PAYG governance policy below.
- **Cost and PAYG governance.** Azure budget alerts at 50%, 80%, 100%; a named cost owner; daily spend review during active build; resource tags for project, agent, capability/slice, owner, and cost owner; no always-on AI/agent runtime without justification; tear down or disable per-capability resources at slice close. Optional Agent 365 is a future governance layer, never a runtime dependency.
- **No manual change to controls.** Gates, authorization rules, and evidence behaviour are changed in facade code under review, never by toggling a portal setting at runtime.

**Referenced policies:** the binding principles (Section 1.2, items 9 and 10), the integration-validation gate (Section 4), the security and privacy controls (Section 12), and the data residency expectations (Section 15). Each capability overlay references these and records its own configuration manifest and approved exceptions.

**Must not:** rely on manual portal configuration that is not recorded; add premium/paid features or provisioned throughput silently; change a control by manual setting rather than reviewed code.

## 17. The shared contract

Every facade endpoint returns the same envelope and status vocabulary, regardless of capability.

**Status vocabulary**

| Status | Meaning | Agent behaviour |
|---|---|---|
| `completed` | Action allowed and performed; state/evidence persisted | Render result and continue |
| `in_progress` | Accepted but not finished; long-running work runs behind the facade (§5). Carries an `operation_id` | Surface a "still working" message and poll the status-poll tool on the `operation_id`; never fabricate a result before a terminal status |
| `blocked` | A workflow gate prevents the action | Display the reason; do not retry or override |
| `needs_input` | Required input missing or ambiguous | Slot-fill the named input, then re-call |
| `validation_failed` | Input failed server-side validation | Show correction guidance; do not bypass |
| `unauthorized` | Caller lacks the required role/scope | Display business-language denial; do not re-decide |
| `error` | Unexpected system failure | Show a stable message; do not fabricate success |

**Standard response envelope (illustrative fields)**

```json
{
  "status": "completed",
  "case_id": "...",
  "operation_id": "...",
  "state_version": "...",
  "evidence_event_id": "...",
  "artifact_refs": [],
  "required_inputs": [],
  "allowed_actions": [],
  "user_message": "...",
  "safe_details": {},
  "correlation_id": "..."
}
```

**HTTP mapping:** every status in the vocabulary — including `unauthorized` — is returned as HTTP 200 with the envelope in the body, so Copilot Studio always receives a renderable outcome (a non-2xx response surfaces to the agent as a tool failure, not an envelope). Non-200 codes are reserved for transport faults the agent cannot meaningfully render: `401` (missing/invalid token, before authorization runs), `409` (optimistic-concurrency conflict; safe to re-read and retry), `500` (unexpected failure; renders as `error`). The authorization decision and its `unauthorized` evidence event are unchanged — only the wire code differs. A correlation ID ties the conversation turn, tool call, evidence event, and telemetry together.

## 18. Data-placement decision guide

With four data technologies in play, place data by its role, not by convenience.

| If the data is... | It belongs in... | Because... |
|---|---|---|
| Authoritative case/work-item state, idempotency, case ACLs | **Azure Table** | It is the system of record; it must be transactional and authoritative. |
| Documents, drafts, exports, evidence payloads | **Azure Blob** | Versioned, never-overwritten file artifacts. |
| The canonical, business-readable audit trail | **Azure Table + Blob (evidence ledger)** | Audit must be immutable and decoupled from runtime telemetry. |
| Contextual memory *contract* (scoped retrieval, save triggers, lifecycle/status, context-pack construction) | **Memory Service / Memory SDK Adapter** | Keeps memory governed and subordinate to authoritative state. |
| Contextual memory *persistence* (summaries, durable facts, proposed facts, scoped context-pack material) | **Cosmos DB** | Flexible, scope-tagged contextual records; accessed only through the memory service/facade. |
| Reference material + embeddings for retrieval | **PostgreSQL + pgvector** | Vector search over grounding material; not authoritative. |
| Operational metrics, traces, safe diagnostics | **App Insights / Log Analytics** | Operational only; never the canonical audit. |
| Secrets and connection material | **Key Vault** | Least-privilege secret handling. |

Two boundaries to hold firm: **Cosmos and Postgres never become the system of record or the evidence ledger**, and **memory and RAG content are always retrieved through the facade and labelled as context, never authority.**

> **Footprint note.** A database-consolidation ADR (evaluating whether the current four-DB footprint — Azure Table + Blob, Cosmos DB, PostgreSQL + pgvector — should collapse onto fewer engines before the pattern hardens) is under evaluation under `Initial_Documentation/adr/`. Whatever it concludes, the binding constraint is the **role separation** above (system of record ≠ memory ≠ RAG ≠ telemetry), which is logical, not physical — it does not require four distinct engines.

## 19. New-capability checklist

When building a new agentic capability, supply a specialization overlay that defines:

1. domain scope and hard boundaries;
2. source-of-truth process documents;
3. roles and permissions (global + case-scoped);
4. topic map (Copilot Studio);
5. tool/action catalogue (facade endpoints);
6. document/source handling rules;
7. generation tasks and which AI backend each uses;
8. whether RAG (Postgres) is needed, and which reference material it indexes;
9. whether memory (the Cosmos-backed Memory Service / SDK Adapter) is needed, with scopes, statuses, and save/retrieve triggers;
10. domain evidence event names (shared naming style);
11. data residency: region of record, approved exceptions, retention windows;
12. configuration manifest and any recorded manual-configuration exceptions;
13. UAT quality threshold and the first functional slice;
14. **Canadian-residency verification — model availability (verify, do not assert):** confirm the chosen model and version is actually deployable in a **Canadian Foundry region (Canada Central / Canada East)**. Not every model/version is available in Canada; a capability designed around an unavailable model forces a **cross-region inference exception** (§15). Record what was verified, when, and against which region — findings are captured per capability overlay (a supporting ADR/checklist doc is being drafted separately under `Initial_Documentation/adr/`).
15. **Canadian-residency verification — Copilot Studio cross-geo data movement (verify, do not assert):** confirm whether the agent's **generative features** move data outside the environment's geography. Copilot Studio generative features in Power Platform environments can involve cross-geo data movement **unless the relevant admin settings restrict it**. Verify the current admin-setting state for the environment rather than assuming; record the finding per capability overlay (same drafted ADR/checklist doc). Both checks change frequently — re-verify, do not rely on a prior assertion.
16. open decisions.

Everything else — the identity model, response envelope, evidence model, storage conventions, logging approach, memory/RAG subordination, security and privacy controls, residency posture, cost controls, manual-configuration policy, tool boundary, and facade responsibilities — is inherited from this document unchanged.

## 20. Anti-patterns to avoid

Each is a defect, not a style preference.

- **UI-driven workflows** — a card or browser handler that calls a mutation endpoint, or front-end logic that sequences business steps.
- **Prompt-only controls** — treating a prompt instruction as an enforced gate, authorization, approval, security, or privacy control.
- **Missing or unstructured audit** — a state change with no evidence event, or free-text logs standing in for the typed evidence model.
- **Treating Copilot Studio as the system of record** — relying on conversation state for trusted business state.
- **Treating memory or RAG as authority** — using memory or a retrieved passage as a gate, approval, candidate status, artifact version, or evidence substitute; or letting one subject's memory influence another's.
- **Bypassing the memory service** — application code, Copilot Studio, or a Foundry backend reading or writing Cosmos memory directly.
- **Making Cosmos, Postgres, or Dataverse the system of record** — authoritative state and evidence stay in Azure Storage.
- **Direct AI-backend calls from Copilot Studio** — any state-changing or business-affecting inference must go through the facade.
- **Power Platform overreach** — placing business rules, gates, or state-changing automation in Power Automate or Dataverse instead of the facade and Azure Storage.
- **Hidden cost growth** — always-on services, provisioned throughput, or silently added paid features.
- **Silent or undocumented manual configuration** — hand-applied portal settings, identity bindings, or residency choices with no record.
- **Cross-boundary data movement** — relocating personal data outside the approved residency boundary without a recorded decision.
- **Skipping the integration-validation gate** — building domain workflow before proving the PAYG/no-premium Copilot-to-Azure path.
- **Per-capability one-off architecture** — inventing a new persistence shape, envelope, or identity model instead of reusing the shared ones.

---

## 21. Summary

The architecture in one line: Copilot Studio (in a Power Platform sandbox) orchestrates; the Azure facade decides and enforces; Azure Storage is the system of record and evidence ledger; Foundry, the Cosmos-backed Memory Service, and Postgres RAG are advisory context behind the facade; Entra, Key Vault, and App Insights carry identity, secrets, and telemetry. Built this way and free of the §20 anti-patterns, every capability is auditable from the first slice, testable without the live model, residency-aware, and cost-governed. See §1.2 for the binding principles and §20 for the defect list.
